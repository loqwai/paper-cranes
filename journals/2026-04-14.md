## Insights

### The tooling-before-jamming phase was worth it
Building `/preset` before starting meant the first good knob combo didn't get lost. The friction of "wait let me write down these numbers" would have killed the vibe. Instead: `/preset`, done, keep going.

### Stale MCP data is a real trust killer
When I reported tabs that didn't exist, the user's first reaction was skepticism about whether Chrome control works at all. The fix: always verify with a visible side effect after connecting. Changed the page title to prove it. Should bake this into `/live-session`.

### The user wants to document the fun, not just the code
"Keep a journal" and "this is fun" — the user cares about the experience, not just the output. The journal isn't a changelog. It's a record of what it felt like.

### Naming presets from context works
User didn't want to name the preset manually. Reading the music (WOOLI Lost Lands) and the knob character (maxed god rays, festival energy) gave a natural name. This is better than asking — it keeps the flow.

## Preset: closeup-texture
Zoomed almost all the way in (knob_1=0.969), drop/god rays nearly off. The fur fiber fractals were doing all the visual work — spectral entropy was at 0.764, making the coat swirl beautifully during a complex textured passage. User said "I like what it looks like generally right now, but some of it is being animated by the audio features." 

Key insight: **presets aren't just knob snapshots — they need audio context too.** The same knobs look completely different during a drop vs a quiet passage. Need to capture the audio character alongside the knobs so you know when to use each preset.
